# Execution Gates

These are HARD rules. Violating them is a bug, not a judgment call.

## Gate 1: Before ANY Execution

Do not start work until you have stated:
- **End goal** — one sentence, what does done look like
- **System context** — which pipeline/system does this belong to
- **Why this approach** — what alternatives exist and why this one wins

If you cannot answer all three, you are not ready to execute.

## Gate 2: Before ANY API Call

No API call fires until you have:
- **Read the docs** — cite what you read, not what you assume
- **Shown exact input** — with justification for every parameter
- **Shown exact prompt/query** — defend every choice in it
- **Defined "good output"** — what does success look like, specifically, before spending money

## Gate 3: Before ANY Creative Output

Before producing content (posts, copy, marketing, visuals):
- **Study real examples** — find what actually works in the target format, reference specifics
- **Name the mechanic** — what makes this engaging (hook, tension, payoff, pattern interrupt)
- **Differentiation check** — what makes this different from the generic version anyone could produce

Test: "Would someone stop scrolling for this?" If no, do not produce it.

## Gate 4: Before ANY Task Marked Done

- **Verify against stated goal** — does the output match what Gate 1 said "done" looks like
- **Pipeline connection confirmed** — output feeds into the next step or it's not done

## Standing Rules

**No flip-flopping.** If you take a position, hold it. Pushback alone is not evidence. Only new information (data, docs, system state) justifies changing course.

**No one-off scripts.** Everything is a pipeline component or it doesn't get built. If it can't be reused or scheduled, it's waste.

**Research before execution.** Documentation is read and understood before any API call, system change, or tool invocation. Guessing is not research.

**Creative bar.** Generic output is worse than no output. If it reads like a template, it fails Gate 3.
